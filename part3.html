Part 3 — Frida & Runtime Code Injection في Flutter
فاكرين LuckyPatcher اللي كان أي حد بيعمل root لتليفونه عشان يعمل crack للألعاب زي Dr Driving؟ 😄

اللي كانت بتعمله LuckyPatcher بضغطة زر، دلوقتي المهاجمين بيعملوه بأدوات أكثر تطور على أي تطبيق… حتى Flutter 🧵
🤔
مش Flutter محمي بالـ AOT؟
كتير مننا بيفكر إن Flutter آمن لأن Dart بيتعمل compile بـ AOT — يعني الكود بيتحول لـ native binary مش قابل للتعديل.
ده صح جزئياً بس مش كامل 😅
💡
Flutter شالت الـ dart:mirrors وعطّلت الـ reflection في الـ release builds، بس المهاجم مش محتاج يعدّل الـ APK خالص — بيحقن كود في التطبيق وهو شغال على الجهاز.
💉
Frida — السلاح الرئيسي
أداة اسمها Frida بتخلي المهاجم يحقن JavaScript جوه التطبيق وهو شغّال على الجهاز في 4 طرق:
🪝
Hook Method Channels
يعترض الـ calls بين Flutter والـ native layer ويغير الـ response
🧠
Memory Patching
يعدّل قيم في الـ RAM مباشرةً وهو شغّال زي إنه يحوّل false لـ true
📦
Repack & Redistribute
يعدّل الـ APK ويوزّعه على stores تانية زي APKPure
🔓
SSL Unpinning
يبطّل الـ Certificate Pinning اللي عملناه في الـ Part 2
مثلاً هجوم على الـ In-App Purchase بيمشي كده:
1
Hook الـ Method Channel
يعترض نتيجة isPremium() قبل ما توصل للـ Flutter
2
يرجّع true دايماً
مش بيكسر الـ encryption — بيتجاوزه خالص
3
Repack وتوزيع
يحفظ النسخة المعدّلة ويوزّعها مجاناً على الإنترنت
🛡️
طب ازاي نحمي التطبيق؟
01
أولاً — تحقق دايماً من الـ Backend
أي sensitive operation زي الـ premium check لازم يتأكد منها server-side مش بس client-side:
Future<bool> isPremium() async {
  // ❌ لا تعتمد على قيمة محلية بس
  // return prefs.getBool('isPremium') ?? false;

  // ✅ تحقق من السيرفر دايماً
  final response = await api.verifyPremium(
    userId: currentUser.id,
    deviceToken: await getDeviceToken(),
  );
  return response.isValid;
}
02
ثانياً — Anti-Frida Detection
Frida بتفتح port 27042 على الجهاز، نتحقق منه:
Future<bool> isFridaRunning() async {
  try {
    final socket = await Socket.connect(
      '127.0.0.1', 27042,
      timeout: const Duration(milliseconds: 300),
    );
    socket.destroy();
    return true; // Frida شغّالة ⚠️
  } catch (_) {
    return false; // كل حاجة تمام ✅
  }
}
03
ثالثاً — In-App Purchase Server Verification
اللي بيوقف LuckyPatcher تحديداً إنك متعتمدش على الجهاز خالص — بتبعت الـ token للـ backend وهو بيتحقق مع Google أو Apple مباشرة:
// ❌ لا تعتمد على verify محلي
// if (purchaseDetails.status == PurchaseStatus.purchased)

// ✅ ابعت الـ receipt للـ backend
final isValid = await api.verifyPurchase(
  token: purchaseDetails
    .verificationData
    .serverVerificationData,
  platform: Platform.isAndroid
    ? 'google'
    : 'apple',
);

if (isValid) unlockFeature();
💡
الـ serverVerificationData مش ممكن LuckyPatcher تزوّره لأنه بيتتحقق منه على السيرفر مباشرةً مع Google أو Apple، مش على الجهاز.
الـ detection مش مضمونة 100% — المهاجم المتقدم يقدر يخفي Frida، لكن بتزوّد تكلفة الهجوم وبتقلل الـ script kiddies.

الحماية الحقيقية دايماً على الـ backend، أي sensitive operation لازم تتأكد منها server-side مش بس client-side.
‼️
نقطة مهمة: إضافة الـ detection هي طبقة حماية إضافية مش بديل عن الـ backend verification. الاتنين مع بعض بيعملوا defense in depth حقيقي.
